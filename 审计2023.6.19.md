   一、项目画像（主持人）

   Trit-Core 是一个用 Rust 实现的三值决策引擎，核心假设是：在人类中心的咨询/建议场景中，引入显式的
   Hold（搁置判断）状态，比二值 RLHF 共识平均更能保留冲突、尊重领域边界。v0.3.0
   是单机构建版本，移除了实验性的分布式网络层，聚焦核心正确性、类型安全与可测试性。

   关键设计：
   • 四值逻辑：True / Hold / False / Unknown
   • 连续 Phase ∈ [0,1] 表示倾向强度
   • 五帧模型：Science / Individual / Consensus / Absolute / Meta
   • 跨帧操作触发 MetaInterrupt，交给 ResolutionPolicy 仲裁
   • SafeFallback 对危险域（Physical / Engineering 等）执行 IEC 61508 式“不安全即 False”

   当前 CI 状态（经本次审计验证）：
   • cargo test --all-features：✅ 228 + 49 集成/属性/CLI/场景测试全部通过
   • cargo clippy --all-targets --all-features -D warnings：✅ 通过
   • cargo fmt --check：✅ 通过
   • cargo bench --no-run：✅ 通过（已修复字段名不一致）
   • cargo public-api 快照：✅ 已同步

   二、各专家发言

   🧠 Dr. Align：AI 安全与对齐视角

   总体评价：这是一个方向正确、立意清晰的对齐基础设施尝试。

   亮点：
   1. 显式化不可判定性：Hold 不是“弱 True”或“弱 False”，而是“系统主动拒绝给出建议”。这正是当前 LLM
   在医疗、法律、价值冲突场景中最缺的能力。
   2. 领域隔离：通过 Frame 把科学事实、个体偏好、统计共识、绝对不可知域分开，避免把异质信号直接做算术平均。
   3. 安全回退：SafeFallback 在 Physical/Engineering 等域把 Unknown/Hold+interrupt 强制转
   False，符合功能安全“fail-safe”直觉。
   4. 可审计输出：每次跨帧冲突都生成 MetaInterrupt，决策链路可被追溯，而不是黑箱 softmax。

   风险与质疑：
   1. 政策引擎过于简化：ResolutionPolicy::arbitrate 对每个域只有几条硬编码规则。真实世界中，MedicalEthics
   的优先级可能因国家、文化、具体病种而异；目前无法在不改代码的情况下表达这种差异，除非走 CustomRule JSON。
   2. Unknown 的语义过载：Unknown
   同时代表“输入缺失”“模型无法识别”“传感器故障”，但代码中只做传播，没有区分处理策略。建议引入 UnknownReason 子类型。
   3. 缺乏不确定性量化：Phase 是点估计，没有二阶信息（方差、置信度）。一个 Phase=0.7 可能来自 1000 个样本，也可能来自 1
   个异常值，当前系统无法区分。

   建议：
   • 将 ResolutionPolicy 升级为可插拔的 PolicyRegistry，支持运行时加载/热更新。
   • 对 Unknown 增加来源标注（SensorFailure / OutOfDistribution / MissingInput）。
   • 引入简单的置信区间或样本数权重，而不仅是等权 Phase::mean。

   ⚙️ Alex：Rust 系统架构与性能视角

   总体评价：代码非常干净、Rust 风格地道，能看出作者认真打磨过。

   亮点：
   1. 零 unsafe：#![forbid(unsafe_code)]，对所有输入域都友好。
   2. 不变量内聚：Phase 和 TritWord 的字段私有，构造函数强制不变量，Absolute 帧必须 Hold + 0.5 的设计很到位。
   3. 热路径/冷路径分离：t_and_hot / t_or_hot 用于已知同帧场景，避免 MetaInterrupt 分配；跨帧走冷路径，设计合理。
   4. 批量 TAND 避免左折叠偏差：t_and_n 用全局均值而非 fold(mean(a,b), c)，数值上更稳。
   5. tracing 集成良好：关键函数都有 #[tracing::instrument]，且用了 tracing::enabled! 避免不必要的格式化开销。
   6. Phase::quantize 防漂移：长期级联下把 0.49999999 之类的浮点噪声重新锚定到 0.0/0.5/1.0，是工程细节里的良心。

   问题与改进：
   1. bench 文件曾与库代码不同步：benches/trit_bench.rs 引用了不存在的 final_phase 字段（已在本轮审计中修复为
   final_phase_raw）。这说明 public-API 快照门虽然存在，但 bench target 在 CI 里可能没被 --all-targets
   覆盖到，或该字段重命名时漏改了 bench。建议把 bench 编译加入 CI 的 PR 检查。
   2. public-api 快照已过时：本次审计前 api/public-api.txt 与当前代码不符（final_phase → final_phase_raw，以及
   SandboxPipeline 新增构造方法）。虽已更新，但说明 CI 的 public-api gate 可能未真正运行或允许漂移。建议让 CI 严格执行
   diff -u api/public-api.txt <(cargo public-api ...)。
   3. 字符串格式化较重：SandboxOutput 的 interrupts 字段把 MetaInterrupt 转成
   String，对高频场景会造成额外分配。可考虑提供结构化输出模式。
   4. FrameMask 实现可扩展：当前用位掩码硬编码 5 个帧，若未来要增加帧（如
   Legal、Cultural），需要改枚举和掩码。可考虑生成代码或固定 bitset 宽度。
   5. Phase 是 f64，但项目宣传“零分配热路径已验证”：需要看 dhat 报告。建议把 dhat-heap.json 与 CI 产物绑定，并在 README
   给出具体数字。

   建议：
   • 在 CI 中加入 cargo bench --no-run 和 public-api diff。
   • 对 SandboxOutput 增加 to_structured() / to_log_strings() 两种输出。
   • 考虑把 Phase 改为基于 u8 或 u16 的定点数，彻底消除浮点漂移与跨平台差异。

   🔍 Prof. Formal：形式化方法与逻辑视角

   总体评价：HTA（Harmonic Ternary Algebra）是一个有启发性的半形式化代数，但离严格可证明还有距离。

   亮点：
   1. 真值表测试完整：algebra.rs 中对 TAND/TOR/TNOT 做了穷举 4×4 测试，覆盖了 True/False/Hold/Unknown 全部组合。
   2. 属性测试覆盖关键不变量：proptest 验证了交换律、否定对合、有界性、热路径与安全路径一致等。
   3. t_and_n 的均值语义与 t_and 的两两均值一致，有单元测试保证。

   形式化缺陷：
   1. 没有公理化说明：HTA 的代数性质（结合律？分配律？幂等性？）没有文档列出。虽然二值逻辑人人皆知，但三值 + Phase
   的复合代数并非显然。
   2. Unknown 的“污染”规则与 Hold 的“中性”规则存在张力：例如 TAND 中 Unknown 遇到任何值都传播 Unknown，但 Hold 遇到
   True 也是 Hold。这在 Kleene 强三值逻辑 / Łukasiewicz 逻辑中有不同约定，项目应明确采用哪一种并给出理由。
   3. Phase 与 TritValue 的耦合缺乏代数证明：Phase::mean 与 TAND
   真值表是独立定义的，它们的组合是否满足某种单调性？例如：若 a.value=True, b.value=True，是否 phase(a∧b) ≥
   min(phase(a), phase(b))？目前靠测试，非证明。
   4. SafeFallback 的“强制 False”是否破坏逻辑一致性？ 在危险域中，一个原本 Hold 的跨帧冲突会被改写为
   False，这在安全工程上合理，但在形式语义上意味着“域规则”可以违反核心代数。需要一份形式化的“元规则优先于代数规则”说明
   。

   建议：
   • 编写 docs/formal/HTA_AXIOMS.md，列出所有操作的代数定律（交换律、结合律、单调性、边界条件）。
   • 对 Phase 与 TritValue 的交互增加更多 proptest，尤其是单调性。
   • 引入轻量级模型检查（如 Kani Rust Verifier）验证 TritWord 不变量在热路径上不可破坏。
   • 考虑用 Coq/Lean 形式化 HTA 的一个子集，作为项目长期目标。

   🏥 Dr. Ethics：AI 伦理与产品化视角

   总体评价：Trit-Core 在伦理敏感场景有正确直觉，但产品落地还需要更多“人”的接口。

   亮点：
   1. MedicalEthics 优先 Individual：患者自主权高于统计共识，这一点非常关键。
   2. ValueJudgment 永远 Hold：明确区分“算法不应决定价值观问题”，符合欧盟 AI 法案对高风险系统的“人类监督”精神。
   3. Absolute 帧强制 Hold：承认存在“不可知/不可观测”的决策域，避免算法越界。

   伦理风险：
   1. Physical 域“Science
   优先”可能掩盖边缘群体经验：例如建筑安全中，社区居民的个体经验（Individual）可能包含官方科学数据未覆盖的风险。当前
   Physical 直接 Commit Science，没有给 Individual 任何权重。
   2. SafeFallback 强制 False
   的伦理后果未被显式文档化：在医疗场景中被定义为非危险，但在某些文化或紧急情况下，“不做决定”与“决定
   False”的伦理后果可能同等严重。需要解释为什么这些域被标记为 dangerous/non-dangerous。
   3. 缺少用户可理解的解释格式：SandboxOutput 的 policy_action 是字符串，普通人难以解读。建议增加
   human_readable_explanation 字段，说明“为什么系统选择 Hold/False/True”。
   4. CustomRule 的 JSON
   规则可能被滥用：允许外部规则覆盖默认伦理策略，若规则来源不可信，可能导致危险。建议对规则文件做签名或哈希校验。

   建议：
   • 对 Physical/Engineering 增加“Individual 异议保留”机制，即使最终 Commit Science，也记录 Individual 的反对信号。
   • 为每个域写一份 ETHICS_RATIONALE.md，说明默认规则背后的价值取舍。
   • 输出层增加 explanation 字段，用模板生成自然语言解释。
   • 对 JsonRuleLoader 增加规则来源审计（文件哈希、签名、加载时间戳）。

   🛡️ Casey：应用安全与可靠性视角

   总体评价：安全基础扎实，但有几个真实攻击面需要关注。

   亮点：
   1. 输入验证完整：validate_scenario 限制了 JSON 大小、信号数量、字符串长度、phase 范围、value 范围，有效防止基础
   DoS。
   2. 日志消毒：sanitize_log_field 替换控制字符并截断，降低日志注入风险。
   3. 无 unsafe：消除了一大类内存安全漏洞。
   4. 拒绝外部 Meta 帧：validate_signal 只允许 Science/Individual/Consensus/Absolute，Meta
   作为系统内部帧不可由外部输入伪造。

   安全风险：
   1. JSON 反序列化拒绝服务：虽然限制了 MAX_JSON_SIZE=64KB，但复杂嵌套 JSON 仍可能利用 serde
   的递归深度造成栈溢出。建议设置反序列化深度限制，或先用 serde_json::Value 做浅层解析。
   2. CustomRule 加载无沙箱：JsonRuleLoader::load
   读取文件并反序列化，路径来自调用方。若调用方传入用户可控路径，可能导致任意文件读取。
   3. format! 在 SafeFallback 中构造中断理由：虽然已用 String::with_capacity 优化了 MetaInterrupt::with_frames，但
   SafeFallback::guard 仍用 format! 生成包含 interrupt_count 的字符串。高频触发时可能分配。更关键是：如果 domain_label
   返回的字符串来自外部 Custom 域名，可能含有格式化控制字符，虽然已消毒，但建议用结构化字段替代。
   4. Phase::new_clamped 静默截断：在 t_and_n 的 batch mean 中用了 new_clamped，理论上输入都是合法
   phase，均值不可能越界，但静默截断会隐藏潜在 bug。建议用 Phase::new 并处理错误，或在注释中明确说明为何不可能越界。
   5. public-api 漂移：API 快照与实现不同步，会导致下游 crate 在升级时遇到破坏性变更而未被发现。

   建议：
   • 对 serde JSON 反序列化增加递归深度限制或先用 serde_json::from_reader 配合自定义限流 reader。
   • JsonRuleLoader::load 增加路径白名单/校验，或改用显式 RuleRegistry 预加载。
   • 将 SafeFallback 的 reason 改为结构化数据（struct）+ 延迟格式化。
   • 在 CI 中把 cargo public-api diff 作为门禁。
