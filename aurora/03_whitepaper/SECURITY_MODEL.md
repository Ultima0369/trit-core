## 一、威胁模型

### 1.1 攻击者分类

| 攻击者 | 能力 | 动机 | 威胁等级 |
|--------|------|------|----------|
| 本地恶意软件 | 访问本地文件系统 | 窃取敏感数据 | 高 |
| 网络攻击者 | 网络嗅探、中间人 | 窃取传输数据 | 中（本地优先降低风险） |
| 物理攻击者 | 盗取设备 | 获取本地数据 | 中（加密降低风险） |
| 用户自身 | 误操作、配置错误 | 数据丢失 | 中 |
| 供应链攻击者 | 污染依赖库 | 植入后门 | 低（Rust 生态相对安全） |

### 1.2 攻击面分析

| 攻击面 | 描述 | 风险等级 |
|--------|------|----------|
| 数据源接口 | 恶意构造的邮件/日历数据 | 中 |
| 用户输入 | 恶意构造的场景 JSON | 低（已有 Trit-Core 输入验证） |
| 网络接口 | 环境数据 API 被劫持 | 低（仅获取公开数据） |
| 文件系统 | 审计日志被篡改 | 中（哈希链保护） |
| 内存 | 敏感数据在内存中残留 | 低（Rust 内存安全） |

## 二、安全设计原则

### 2.1 本地优先 = 攻击面最小化

- 无网络监听端口
- 无持久化远程连接
- 无多用户认证（单用户系统）
- 数据不离开设备，除非用户明确导出

### 2.2 数据加密

- **静态加密**：SQLite 数据库使用 AES-256-GCM 加密
- **密钥管理**：密钥由用户持有，系统不存储明文密钥
- **内存保护**：敏感数据在内存中使用后立即清零（`zeroize` crate）
- **导出加密**：导出数据时可选加密，密码由用户设置

### 2.3 输入验证

- 所有外部输入经过严格验证（大小、类型、范围）
- 路径遍历防护（CWE-22）
- JSON 反序列化炸弹防护（大小限制 64KB）
- 非法 Phase 值拒绝（`Phase::new` 返回 `Err`）

### 2.4 审计与不可篡改

- 审计日志使用哈希链（Hash Chain）
- 每行包含：前一行的 hash + 当前内容 + 当前 hash
- 篡改检测：验证哈希链完整性

```
Row 1: data="init" | hash=SHA256(data)
Row 2: data="op1" | prev_hash=Row1.hash | hash=SHA256(prev_hash + data)
Row 3: data="op2" | prev_hash=Row2.hash | hash=SHA256(prev_hash + data)
```

## 三、缓解措施

| 风险 | 缓解措施 | 状态 |
|------|----------|------|
| 数据泄露（设备被盗） | AES-256-GCM 加密，密钥用户持有 | 已实现 |
| 恶意输入 | 输入验证、大小限制、类型检查 | 已实现 |
| 日志篡改 | 哈希链追加写入 | 已实现 |
| 内存泄露 | Rust 内存安全、zeroize | 已实现 |
| 依赖漏洞 | cargo-audit、定期更新 | 持续 |
| 社会工程 | 产品教育、明确权限 | 持续 |

## 四、安全审计计划

- **M0**：代码审查（self-audit）
- **M1**：第三方安全审计（渗透测试）
- **M2**：安全认证（ISO 27001 或 SOC 2 准备）
- **M3**：企业级安全审计（客户要求）
