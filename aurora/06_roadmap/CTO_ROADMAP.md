## 一、产品定位（一句话）

> **Aurora 是一个注意力主权训练系统。它以"提醒注意力指向"为起点，通过 Frame 标注、三值决策、节律分析、注意力调度等技术，训练用户不被现象劫持，并提供可检验的训练效果。**

### 1.1 与传统效率工具的区别

| 维度 | 传统效率工具 / AI 助手 | Aurora |
|------|----------------------|--------|
| 目标 | 帮用户更快做更多事 | 帮用户看见自己的注意力被什么捕获 |
| 对冲突的态度 | 消抹冲突，给出"最佳"答案 | 保留冲突，输出 Hold |
| 与用户的关系 | 越懂你越好 | 越不操控你越好 |
| 成功指标 | DAU / 留存 / 任务完成数 | 注意力自主性提升（可检验） |
| 最终交付 | 用户离不开产品 | 用户不再需要产品 |

### 1.2 核心用户价值

1. **觉察**：用户能实时看到自己的注意力指向哪个 Frame（身体、角色、社交、环境等）。
2. **悬置**：当多个 Frame 冲突时，系统不替用户决定，而是提醒"这里需要悬置"。
3. **训练**：通过 AttentionScheduler 的 ShiftTo / HoldCurrent / Recalibrate 建议，训练用户主动调度注意力。
4. **检验**：通过节律分析（基频稳定性、相位恢复速度）验证训练是否有效。

## 二、北极星指标与反指标

### 2.1 北极星指标

> **注意力自主性指数（Attention Sovereignty Index, ASI）**

定义（M0 简化版）：

```
ASI = (用户主动调度次数) / (系统提醒次数) × (相位恢复速度系数)
```

- **用户主动调度次数**：用户在收到提醒后，执行了 ShiftTo / HoldCurrent 选择的次数。
- **系统提醒次数**：系统检测到注意力被现象劫持并发出提醒的次数。
- **相位恢复速度系数**：环境冲击后，用户基频回到基线的速度（相对于历史基线）。

ASI 越高，说明用户越能自主管理注意力，系统越成功。

### 2.2 阶段性指标

| 阶段 | 指标 | 目标 |
|------|------|------|
| M0 | 提醒准确率 | 跨 Frame 冲突检测准确率 > 95% |
| M0 | 基频检测误差 | 合成数据 < 1% |
| M1 | 主动调度率 | 用户收到提醒后主动调度 > 30% |
| M1 | 相位恢复速度 | 相比未使用系统时提升 > 10% |
| M2 | 角色入侵识别率 | 模拟角色入侵场景 > 90% |
| M2 | 环境冲击预警提前量 | > 1 小时 |
| M3 | 团队级相变检测延迟 | < 5 分钟 |
| M4 | 企业部署数据主权合规率 | 100% |

### 2.3 反指标（必须抑制）

以下指标上升，说明产品背叛了定位：

- **用户对系统建议的依赖率**（应该下降）
- **系统打开时长 / 日均使用次数**（不应追求）
- **用户画像维度数**（禁止增长）
- **云端数据同步频率**（应尽可能低）
- **推送通知频率**（应仅在关键时刻触发）

## 三、技术架构：注意力训练闭环

```
┌─────────────────────────────────────────────────────────────────┐
│                        现象层（Phenomena）                        │
│  邮件元数据 · 日历事件 · 通知时间 · 位置变化 · 生理信号 · 环境数据    │
│  原则：只采元数据，不读内容；本地存储；用户可随时关闭任一数据源       │
└─────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        感知层（Perception）                       │
│  通信频率时间序列 → 小波分析（CWT/DWT）                            │
│  输出：基频 f₀ · 谐波 · 相位漂移 Δφ · 频谱重构 DKL                  │
│  输出：环境冲击等级 · 角色边界渗透指数                              │
└─────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        标注层（Annotation）                       │
│  Embodied（身体）· Individual（个人）· Relational（关系）            │
│  Role（角色）· GeoEco（地理生态）· Environmental（环境）             │
│  Phase：注意力倾向程度 [0.0, 1.0]                                  │
│  Confidence：信噪比 / 小波功率 / 数据完整性                         │
└─────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        运算层（Computation）                      │
│  Trit-Core：TAND / TOR / TNOT / 跨域冲突检测                        │
│  ResolutionPolicy：按 Domain 仲裁（Organizational / Relational 等） │
│  SecurityMode：Service / Refusal / Awareness / Transparency        │
│  AttentionScheduler：ShiftTo / HoldCurrent / Recalibrate           │
└─────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        反馈层（Feedback）                         │
│  提醒："你的注意力当前被 [Role] 捕获"                              │
│  训练：提供 ShiftTo / HoldCurrent 选项，用户选择                    │
│  记录：用户覆盖 / 采纳 / 忽略，全部写入审计日志                      │
│  检验：对比训练前后的节律稳定性与相位恢复速度                        │
└─────────────────────────────────────────────────────────────────┘
                                ↓
┌─────────────────────────────────────────────────────────────────┐
│                        呈现层（Presentation）                     │
│  注意力图谱（雷达图）· 冲突面板 · 节奏报告 · 决策审计                 │
└─────────────────────────────────────────────────────────────────┘
```

### 3.1 关键设计约束

1. **本地优先**：所有数据默认本地处理，不上云。
2. **元数据优先**：不读取邮件内容、不读取聊天记录、不读取文档内容。
3. **不进化**：系统出厂设置不变，不基于用户数据学习。
4. **可覆盖**：任何系统输出，用户都可以忽略或覆盖。
5. **可断开**：用户可随时导出、删除、卸载，系统不挽留。

## 五、工程学纪律

### 5.1 代码纪律

1. **伦理门禁测试不可跳过**
   ```bash
   cargo test ethics_
   ```
   失败 = 禁止发布。

2. **无 unsafe 代码**
   Aurora crate 使用 `#![forbid(unsafe_code)]`。

3. **CI 流水线**
   ```
   cargo fmt --check
   cargo clippy --workspace --all-targets --all-features -- -D warnings
   cargo test --workspace --all-features
   cargo test ethics_
   cargo audit
   cargo build --release
   ```

4. **ADR 驱动变更**
   - 修改 CHARTER / FIRST_PRINCIPLES 核心内容：禁止。
   - 修改 Frame / Domain enum：需要 ADR。
   - 新增数据源：需要 ADR。
   - 新增网络依赖：原则上禁止，除非 ADR 特批。

5. **文档同步**
   - 修改代码 → 同步 `map/06_code.md`
   - 新增概念 → 同步 `map/` MOC 和 `docs/`
   - 新增 ADR → 同步 `map/03_adr.md`

### 5.2 设计纪律

1. **任何功能必须有关闭开关**
   - 用户可关闭任何数据源、任何提醒、任何安全检查。
   - 关闭行为本身被记录，但不阻止。

2. **Awareness 不阻断**
   - `SecurityMode::Awareness` 下，代码中不允许 `return Err` 或 `panic!`。
   - 只允许返回 `MetaInterrupt` 通知。

3. **Hold 时不猜测**
   - 当系统输出 `Hold` 时，不允许提供"最可能"答案或概率排序。

4. **不基于用户数据学习**
   - 禁止 RLHF、禁止微调、禁止用户画像、禁止 A/B 测试。

### 5.3 发布纪律

每次发布前必须：

1. 完成 `TECH_REVIEW_CHECKLIST.md` 全部检查项。
2. 由至少 1 名开发者逐项勾选并签名。
3. 任何未勾选项必须有 ADR 记录。
4. 发布版本号遵循 SemVer。
5. 发布说明中包含伦理门禁测试结果。

## 七、风险与缓解

| 风险 | 概率 | 影响 | 缓解 |
|------|------|------|------|
| 小波分析性能/精度不足 | 中 | 高 | M0 用 FFT 证明概念；M1 引入 CWT；持续用合成数据验证 |
| 用户不理解 Hold 设计 | 高 | 中 | 文档强调"Hold 不是失败"；UI 明确解释冲突结构 |
| 本地优先限制功能扩展 | 中 | 中 | 用 Tauri + 本地 SQLite；团队版才引入可选本地服务器 |
| 伦理约束限制商业化 | 中 | 中 | 定位高端用户；不卖数据；一次性买断 + 订阅并行 |
| 数据源格式频繁变化 | 高 | 中 | 抽象 DataSource trait；JSON fallback 作为保底 |
| 安全审计发现漏洞 | 低 | 极高 | M2 引入第三方审计；M0-M1 用 cargo-audit + 属性测试 |
