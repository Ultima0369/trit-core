# 认识论谦逊声明 — 项目使用提醒

**Version**: 0.3.0
**Status**: Active

---

## 核心提醒

> 本项目全部内容，属于**提醒**，而并非**指教**。
>
> 我们邀请每一位学者亲自实践检验，并保持同样的科学成果参考、交叉验证，保持开放与探索，而不是陷入任何一种断言。
>
> 是非对错可以有，不得不如此；但面对具体事物，最好审计情景是否匹配、变量采集是否已足够，然后再有判断。这将可以大大避免陷入草率、肤浅，最后指导了错误的行为。

---

## 为什么需要这份声明

Trit-Core 是一个工程化的三值决策框架，它试图在算法层面回答一个困难问题：**当冲突来自不可通约的参考系时，系统如何诚实地表达“我不知道”？**

但任何框架都是**临时约定**——它在特定约束、特定尺度和特定目的下有效。把它从“提醒”误读为“指教”，意味着把一种情境下的有效规则，不加审计地推广到所有情境。这正是 Trit-Core 试图避免的二值化暴力：

- 把统计共识当成真理
- 把多数决当成正义
- 把工程规则当成伦理答案
- 把“在特定条件下有效”当成“ universally 正确”

因此，这份声明不是谦虚的姿态，而是项目的方法论底线。

## 七项原则

### 1. 提醒，而非指教

Trit-Core 提供的是：
- 一个可运行的三值决策引擎
- 一组经过测试的冲突处理规则
- 一套可审计的元中断机制
- 一种把 Hold 视为一等状态的工程语言

它不提供：
- 对生活具体问题的现成答案
- 替代医学、法律或伦理专家判断的权威
- 适用于所有文化的价值排序
- 可以把“应当如何生活”外包给算法的许可

### 2. 亲自实践检验

所有场景保存为 JSON，所有决策可通过以下命令复现：

```bash
cargo test --all-features -- --test-threads=2
cargo run --bin trit-sandbox -- --scenario scenarios/<name>.json --trace --diagnostic
```

不要只读文档。运行它。修改场景。观察输出何时 Hold、何时 Commit、何时触发 SafeFallback。只有亲自检验，才能判断它在你关心的情境中是否有用。

### 3. 科学成果参考与交叉验证

Trit-Core 的论述借鉴了多个学科：
- **逻辑学**：三值逻辑、直觉主义逻辑、哥德尔不完备性
- **神经科学**：髓鞘化、内感受、神经可塑性
- **热力学**：信息论效率与能耗约束
- **安全工程**：IEC 61508 fail-safe 原则
- **生态学**：多样性、正反馈、系统韧性
- **现象学/认知科学**：具身认知、第一人称体验

但这些引用都是**启发性的**，不是**证明性的**。每一项都需要在你具体的问题域中重新验证。

### 4. 开放与探索，而非断言

项目文档中会出现一些看起来很强的表述，例如：
- “心智是三值运算的”
- “统计共识 ≠ 真知”
- “二元系统在数学上无法表达不可比较”

这些表述应被理解为**工作假设**和**工程启发**，而非封闭的哲学定论。它们的价值在于激发检验，而不是终止讨论。

### 5. 是非对错可以有，但要看前提

Trit-Core 并不否认存在对错。在 `Physical` 域，桥梁疲劳数据说“不安全”， consensus 投票不能改变它；在 `Engineering` 域，安全系数必须被尊重。

但这些“对错”的有效性依赖于：
- 参考系是否匹配（Frame 是否一致）
- 测量是否可靠（Phase 是否合理）
- 域规则是否适用（Domain 是否选择正确）
- 约束是否被满足（输入是否经过验证）

一旦前提改变，结论就可能改变。这就是“审计情景是否匹配”的含义。

### 6. 审计情景匹配度

在把 Trit-Core 应用于任何真实决策之前，请问：

| 审计问题 | 为什么重要 |
|---|---|
| 这个决策真的涉及跨域冲突吗？ | 如果所有信号同 Frame，二元逻辑可能足够 |
| 信号的 Frame 标注准确吗？ | 错误的 Frame 标注会导致错误仲裁 |
| 选择的 Domain 与决策性质匹配吗？ | 用 Physical 规则处理 ValueJudgment 是范畴错误 |
| 是否存在不可通约的价值？ | 如果有，输出 Hold 是诚实的，不是失败 |
| 危险域是否已正确标记？ | SafeFallback 只在 dangerous domain 触发 |

### 7. 变量采集充分性

Trit-Core 的输出质量取决于输入质量。如果变量采集不足，即使逻辑正确，也会给出误导性结果。请检查：

- 是否遗漏了关键信号来源？
- Phase 是否反映了真实的倾向强度？
- 是否存在 Unknown 信号未被识别？
- 环境动态是否已纳入考虑？
- 个体实情是否被压缩为统计平均？

## 对研究者的建议

1. **把 Trit-Core 当作一个可证伪的假设集合**，而不是一个理论体系。
2. **在你的领域中测试它**：医学、法律、教育、公共政策、工程设计——每个领域都有自己的 Frame 和 Domain。
3. **记录反例**：如果 Trit-Core 在你的情境中给出了明显错误的结果，那就是最有价值的反馈。
4. **保持跨学科对话**：不要把工程规则当成哲学答案，也不要把哲学批评当成工程无能。

---

## 对使用者的建议

1. **不要让算法替你做不可通约的价值判断**。`ValueJudgment` 域永远返回 Hold，这正是系统在说：“这个问题我不该回答。”
2. **在危险域中信任 SafeFallback，但审计它的触发条件**。它的设计是“不能决定就意味着不做”，但这不意味着你可以放弃情境判断。
3. **把输出当作决策支持，而非决策本身**。MetaInterrupt 日志告诉你哪里存在冲突，但冲突的解决权属于人类。

## 常见误读与澄清

| 误读 | 澄清 |
|---|---|
| “Trit-Core 认为科学不重要” | 否。`Frame::Science` 在 Physical/Engineering 域具有最高优先级。项目反对的是把科学共识或统计共识升格为个体真理。 |
| “Trit-Core 是反 AI 的” | 否。它是 AI 对齐的一种替代方案，主张在不可对齐的域中保持诚实，而不是强行平均。 |
| “Hold 就是逃避决策” | 否。Hold 是积极的认知操作：它标记冲突、保留可能性、防止草率结论。在某些域（如危险域），Hold 会触发 SafeFallback，转化为明确的 False。 |
| “Trit-Core 给出了生活答案” | 否。它是一个提醒系统，告诉你何时需要人类判断，而不是替你做出人生选择。 |

## 结语

Trit-Core 最终是一个关于**诚实**的工程尝试：

- 对系统自身知识边界的诚实
- 对个体实情的诚实
- 对不可通约价值的诚实
- 对科学方法边界的诚实

这种诚实不是软弱，而是防止技术系统以“中立工具”之名，行“价值强加”之实的前提。

如果您在阅读或使用本项目时发现任何表述被误读为“指教”，请把它拉回“提醒”——并亲自检验它是否适用于您的具体情境。
