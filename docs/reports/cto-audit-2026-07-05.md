## 第一部分：架构全景

### 真实层级

```
                    ┌──────────────────────────────────┐
                    │  ui/ (React + TypeScript + CesiumJS)  │  ← 展示层
                    ├──────────────────────────────────┤
                    │  src-tauri/ (Tauri shell)              │  ← IPC 桥
                    ├──────────────────────────────────┤
   aurora crate     │  app.rs → pipeline/{analysis,attention} │  ← 编排层
   ─────────────    │  bc/ (6 个 Bounded Contexts)           │  ← 业务逻辑
                    │  percept/ (Cloud/LLM → Local/LLM → FFT)│  ← 感知降级链
                    │  db/ (SQLite + schema migration)       │  ← 持久化
                    ├──────────────────────────────────┤
   trit-core crate  │  core/ (TritValue, Phase, Frame,       │
                    │         TritWord, TernaryAlgebra)       │  ← Layer 4: 代数引擎
                    │  meta/ (Domain, ResolutionPolicy,      │
                    │         MetaInterrupt, SafeFallback,    │
                    │         FrameMask)                      │  ← Layer 4: 策略引擎
                    │  anchor/ (5 个稳态约束)                 │  ← Layer 1: 否决权
                    │  hook/ (ScenarioRecognizer,            │
                    │         MountArbiter, HookContext)      │  ← Layer 2: 调度
                    │  adapters/ (10 个认知模块)              │  ← Layer 3: 模块池
                    │  feedback/ (proxy_env, correction)      │  ← Layer 5: 反馈
                    │  security/ budget/ calibration/ clock/  │  ← 支撑模块
                    │  sandbox/ (Pipeline, I/O, validation)   │  ← 集成层
                    └──────────────────────────────────┘
```

**真实数据流**（与 CLAUDE.md 一致）：
```
JSON scenario → ScenarioInput → validate → TritWord[] → t_and_n (batch TAND)
    → MetaInterrupt[] → ResolutionPolicy::arbitrate() → SafeFallback::guard()
    → SandboxOutput (JSON)
```

### 三种路径

1. **热路径（~80%）**: 同帧 TAND → Phase 算术平均 → 无 MetaInterrupt 分配 → 直接输出
2. **冷路径（~20%）**: 跨帧 TAND → MetaInterrupt 生成 → ResolutionPolicy 仲裁 → SafeFallback
3. **感知路径**（Aurora 新增）: FFT 小波分析 → TritWord 映射 → DecisionEngine → AttentionScheduler → SQLite 审计

## 第二部分：按模块深度审查

### 2.1 核心代数引擎（src/core/） — 评分 A

**优点**:
- `TritValue` 使用 LUT 实现无分支 `negate()`、`to_i8()`、`discriminant()` — 正确的热路径优化
- `Phase` 私有化 `f64`，严格构造器 `new()` + 宽松 `new_clamped()` 双入口 — 区分"这是 bug"和"这是脏数据"
- `TritWord` 的三个构造器形成信任梯度：`new()`（信任调用者）+ `try_new()`（不信任输入）+ `from_parts()`（需验证）
- `Frame::Absolute` 不变量在 `from_parts()` 中运行时强制（非 debug_assert）
- `TernaryAlgebra::t_and_n()` 正确使用等权 Phase 平均，避免左折叠偏差
- `SensorSignal`、`BodyState`、`CogState`、`EnvSnapshot` 类型 —— 为多模态感知预留了正确的类型桩

**需改进**:
- 13 处 `#[cfg(test)]` 的 `.unwrap()` 在测试代码中可接受（测试应该 panic），但生产代码也有 268 处 `.unwrap()`。大部分在构造 Phase/TritWord 时，因为调用者已验证值在范围内，但切换到 `.expect("msg")` 会提高可调试性
- `HoldState` / `HolderConfig` 定义了但仅在 `DecisionEngine` 和 `SandboxPipeline` 中有 token 引用 —— 概念正确但尚未在 pipeline 路径中实际使用

### 2.2 策略引擎（src/meta/） — 评分 A-

**优点**:
- `FrameMask` 的 `u16` 位掩码 O(1) 帧存在性检查 —— 正确的微观优化，12 帧刚好装进 u16
- `ResolutionPolicy::arbitrate()` 域分发清晰：Physical/Engineering 优先 Science，MedicalEthics 优先 Individual，ValueJudgment 永远 Hold
- `SafeFallback` 遵循 IEC 61508 语义：危险域 + 不确定性 → 强制 False
- `FallbackBehavior` 使用类型安全枚举取代字符串 —— 消除了一整类拼写错误 bug
- `MetaInterrupt` 的 `with_frames()` 使用预分配字符串构造避免 `format!()` 开销

**问题**:
- **P2**: `JsonRuleLoader` 存在且可工作，但未在 pipeline 的任何生产路径中集成。`CustomRule` / `Custom(String)` 的端到端测试仅存在于单元测试中
- **P2**: `arbitrate_general()` 的近中性检查（`1e-3` 阈值）是合理的启发式，但硬编码。应可配置
- **P2**: `TritValue::From<i8>` 静默将范围外值（如 2、-2）映射为 `Hold`，不报错。`from_i8_strict` 存在但不是默认——调用者可能永远不知道自己传了非法值
- **P2**: `validate_signal()`（`src/sandbox/validate.rs:81-88`）硬编码仅接受 7 个 Frame 字符串。`GeoEco`、`Developmental`、`Role`、`Environmental` 四个 Frame 变体存在、可解析，但不能作为外部场景信号输入——文档未说明此限制
- **P2**: `MetaInterrupt::frames()` 通过解析 reason 字符串提取帧名——脆弱的字符串解析。若格式变更，静默返回 `("Unknown", "Unknown")`。帧对应存储为字段而非从展示字符串解析
- **P3**: `TritWord::new()` 对 Absolute 不变量的检查使用 `debug_assert!`——release 构建中 `TritWord::tru(Frame::Absolute)` 静默接受无效状态。`tru()`/`fals()`/`hold()` 等便捷构造器都走 `new()`，不走 `from_parts()`

### 2.3 适配器层（src/adapters/） — 评分 B+

**结构正确**: 10 个模块各实现 `CognitiveModule` trait，通过 `HookContext` 通信，不互相调用。`adapter_lifecycle!()` 宏精简了样板代码。

**问题**:
- **P1**: 10 个适配器中，最少 3 个在当前 pipeline 路径中从未被 MountArbiter 挂载或调用。`CognitiveDeconstruction`（313 行）、`ConflictSuspension`（153 行）、`CouplingAdapter`（126 行）、`EcologicalAssessment`（141 行）——全部完整实现、有测试、通过编译，但从未被 `mount_arbiter.rs` 的任何 ScenarioType 分支引用。如果它们对当前版本不必要，就应该删除或归档。如果必要，就应该挂载。
- **P3**: 10 个 ponytail 注释（`// ponytail: on_mount + state via adapter_lifecycle_no_unmount!()`）标记了宏生成的生命周期。这本身是好的文档化，但 10/10 模块都以完全相同的方式使用宏，意味着宏可能过度泛化了

### 2.4 沙箱管线（src/sandbox/） — 评分 B+

- `SandboxPipeline`（715 行）过重。`DecisionEngine` 已从中抽取（正确的方向），但 pipeline 尚未完全使用 `DecisionEngine`——它仍在自己的 `run()` 方法中直接调用 `TernaryAlgebra`。
- **P2**: `PhaseTracker` / `Trend` / `PhaseTracker::trend()` 在 `phase.rs` 中实现并测试，但在 `SandboxPipeline` 中未连接。`trace_phase: bool` 字段存在，但没有代码读取 PhaseTracker 输出的 trend
- **P2**: `build_decision_preview()`（`src/anchor/mod.rs:252-280`）将 `ambient_arousal * 1e6` 映射为 `expected_energy_joules`，`ambient_arousal * 1e3` 映射为 `expected_carbon_kg`——这些乘数完全没有物理依据。五个锚约束的所有 `check()` 方法要么检查硬编码配置值，要么检查这个合成 `DecisionPreview`。锚层的"否决权"建立在合成数据之上
- **P2**: `FlourishingPool` 用硬编码值初始化（Autonomy 0.5/0.7、Creativity 0.5/0.6 等），运行时从未被更新——是死状态
- `SandboxDiagnostics` 结构完善，包含 timing、interrupt_count、anchor_check_count 等 —— 生产可观测性的正确基础

### 2.5 Aurora 管线与 BC 层（aurora/src/） — 评分 B

**优点**:
- BC 边界清晰：6 个 BC 各自暴露一个 trait，有明确的 aggregate root
- SQLite schema 迁移系统正确实现
- `run_analysis()` + `run_attention()` 两条独立链路，编排清晰
- 伦理门禁测试（10 项）对应 CHARTER.md 的 4 条底线

**高优先级问题**:

- **H1**: `Database::clone()` 对内存数据库创建空独立 DB（`db/mod.rs:85-96`）。注释承认了这个问题，但没有运行时守卫。任何人切换到 `:memory:` 模式都会遭遇静默数据丢失
- **H2**: `AuroraApp` 使用 `Arc<Mutex<Database>>`，每个管线调用都 `lock().clone()`——每次打开新的 WAL 连接。在高频 Tauri 轮询下造成连接泄漏压力
- **M3**: `pipeline/analysis.rs` 中的 `PhaseTrajectory` / `TrajectorySummary` / `trend()` / `velocity()` 全部实现并测试通过，但在 `AuroraApp` 中从未实例化或 `.update()` 调用——完整的死代码
- **M4**: `BcError` 携带类型化变体（NotFound、InvalidInput、InvalidState、Domain、Io），但在 `app.rs` 中立即被擦除为 `anyhow::Error`，在 `commands.rs` 中进一步擦除为 `String`。三层类型擦除意味着前端永远无法在程序层面区分"未找到"和"I/O 错误"

### 2.6 感知层（aurora/src/percept/） — 评分 C+

**设计正确但实现粗糙**:

- **M1**: 每个 LLM provider 嵌入独立 `tokio::runtime::Runtime`（`cloud.rs:94-97`）。`AuroraApp::new()` 构建两个 provider 并持有它们的 runtime 整个应用生命周期——两个闲置 runtime 占用线程
- **M2**: `PerceptChain` 只尝试第一个可用 provider，无重试逻辑，无按 provider 超时区分。Cloud LLM 有 30 秒超时，用户需等待 30 秒才能降级
- **L3**: `FFTProvider::perceive()` 返回空 `PerceptBatch`——它是最底层"永不离线"的安全底线，但实际上对降级链没有任何贡献
- **L4**: 云 LLM 的 Frame 解析（`cloud.rs:241-250`）只接受 12 个 Frame 中的 4 个（Science、Individual、Consensus、Absolute）。Embodied、Relational、GeoEco、Developmental、Role、Environmental、FirstPerson、Meta 全部静默丢弃

### 2.7 桌面壳（src-tauri/） — 评分 B

**扎实的工程**:
- 两级瓦片缓存（L1 内存 LRU + L2 磁盘）
- actix-web 代理服务器（127.0.0.1:21337）
- 多 CDN 资产预下载 + 故障转移
- 严格的 CSP 头配置
- IPC 契约（TypeScript → Rust）前后端一致

**关键缺口**:
- **I1**: ~2,500 行 Rust，零测试。lib.rs 编排了复杂的启动序列（logger init、数据目录解析、L1/L2 缓存创建、actix-web 代理启动、后台资产下载、AuroraApp 初始化、Tauri builder 设置）——多个故障模式，全部未测试

### 2.8 前端（ui/） — 评分 B+

- React 18 + TypeScript + Vite，类型严格
- CesiumJS 3D 地球 + MapLibre 2D 地图双视图
- `BootScreen` 质量门禁（MSC + 安全 + 架构检查）
- `DecisionDrawer` 显示 ASI 仪表、Frame 张力条、冲突列表——概念正确
- `ErrorBoundary` 正确实现
- 组件偏大（`Earth.tsx` 可能超过 400 行），状态管理分散在多个 `useState` 调用中，没有集中式 store

## 底部评价

**这不是一个玩具项目。** 三元代数引擎（TritValue、Phase、Frame、TritWord、TernaryAlgebra）达到了生产级的类型安全和不变性强制执行标准。19 项基于性质的测试证明了作者理解代数验证。12 个对抗性场景显示了认真的安全思维。

**但它在集成层暴露了研究原型特征。** 感知链虽然架构合理，但实现粗糙（独立 tokio runtime、静默 Frame 丢弃）。桌面壳虽然有出色的缓存架构和严格的安全配置，但完全未经测试。愿景文档（MONOLITHIC_BOT_SPEC）和未连接的代码（PhaseTrajectory、多个适配器）表明项目在概念阶段迭代迅速，需要一次"修剪"通行来对齐文档与代码。

**核心风险**: DB clone 陷阱（H1）和 WAL 连接泄漏（H2）如果部署到有真实用户的多会话环境，会导致数据损坏。

**核心优势**: 三元逻辑代数正确、不变量强制执行到位、测试覆盖真实、CI 通过、安全思维贯穿全栈。这比大多数研究项目好一个数量级，比大多数生产项目差约 6 个月的打磨。
